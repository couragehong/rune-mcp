RFC #85 · v0.4.0 · 2026-04 · Go 1.25

Rune: Python MCP → Go runed 데몬 마이그레이션

현재 Python FastMCP 플러그인(mcp/server/server.py 2000 LoC + agents/* 6000 LoC)을 Go 1.25 기반 상주 runed 데몬 + 얇은 rune CLI 구조로 단일 cutover 전환합니다.

작성일 2026-04-17 · 팀 리뷰용 · 코드베이스 기준 검증 완료

Go 1.25 baseline — 채택한 신규 기능
  • Go 1.22+: loop var per-iteration binding (더 이상 i, q := i, q 불필요) · http.ServeMux method+path routing (mux.HandleFunc("POST /capture", h))
  • Go 1.23+: timer/ticker GC 즉시 회수 (정기 리셋 패턴 더 안전) · iter.Seq(capture_log 스트리밍)
  • Go 1.24+: os.Root(~/.rune/ 경계 안에서만 파일 열기 — symlink traversal 차단) · crypto/hkdf·crypto/sha3 stdlib · generic type aliases · weak.Pointer
  • Go 1.25: testing/synctest(가짜 시계로 fsnotify debounce·circuit breaker·retry backoff 결정적 테스트) · sync.WaitGroup.Go()(Add/Done 보일러플레이트 제거) · runtime/trace.FlightRecorder(크래시/panic 시 직전 N초 trace 덤프) · container-aware GOMAXPROCS (cgroup v2 CPU quota 자동 인식 — systemd CPUQuota와 자연 정렬) · optional GOEXPERIMENT=greenteagc (tail latency 개선 실험)

go.mod: go 1.25 + toolchain go1.25.0 고정. CI matrix는 1.25.x만 — 1.24 호환은 목표 아님.

요약 (Executive Summary)

왜 바꾸는가: 세션당 Python 프로세스 × 모델 × gRPC 채널 × 키 중복이 선형 비례로 증가하는 구조. 상주 데몬으로 바꾸면 세션 수에 무관하게 상수 자원으로 수렴하는 정성적 이득이 기대된다.

전송 레이어 두 가지 안 (Go 공통 데몬 위에서)

안 A — MCP + CLI + runed (3 바이너리)

  • 세션당 rune-mcp shim 1개 상주 (경량 — 실측 ⑩.2)
  • stdio MCP ↔ HTTP 변환기. 에이전트 md 수정 불필요
  • MCP JSON Schema로 파라미터 구조 강제 가능
  • 기존 Claude Code MCP 인프라 재활용

안 B — CLI + runed (2 바이너리, RFC 본문)

  • 세션당 상주 프로세스 없음. rune CLI는 호출 때만
  • 에이전트 md에서 Bash exec으로 rune capture 호출
  • 에이전트 md Step 3(호출부)만 재작성 — 약 10%
  • 배포·디버깅 단순 (curl로 소켓 찔러보기 가능)
중요: runed 코드 자체는 두 안에서 100% 동일. 차이는 HTTP 요청이 소켓에 도달하기까지의 얇은 껍질(~10-15% LoC)뿐. 안 선택은 Phase 1 이후에도 변경 가능.

7.2.4 메모리 leak 방지 체크리스트

  • 모든 io.Closer는 defer Close() — HTTP body, gRPC stream
  • goroutine leak: 모든 go func()에 ctx 전파, 상위 cancel로 정리
  • context.WithTimeout 필수: HTTP handler 30s / gRPC 10s / embed 5s
  • CI에 go.uber.org/goleak로 테스트 종료 후 남은 goroutine 0 검증
  • /debug/pprof/ endpoint (build tag gate: //go:build debug, dev 빌드에서만 컴파일 포함, unix socket 내부로 한정) → heap/goroutine live profiling. 상세는 #50
  • agent_dek 사용 후 best-effort zeroize: for i := range dek { dek[i] = 0 }; runtime.KeepAlive(dek) — KeepAlive 없으면 컴파일러가 앞선 루프를 "dead store"로 제거할 수 있음. 단, escape analysis로 heap 할당 가능성 있으므로 GC 사이클 완료까지는 메모리에 남을 수 있다는 한계 인지 (hard guarantee 아님)

7.3.1 Recall 병렬화 — 가장 큰 승리

// Python: expanded_queries[:3] 순차 → 3× RTT
// Go 1.25: errgroup으로 병렬 → HTTP/2 multiplexing으로 단일 채널 재사용
// (Go 1.22+ loop-var per-iteration binding으로 `i, q := i, q` shadow 불필요)

g, gctx := errgroup.WithContext(ctx)
results := make([][]SearchResult, 3)
errs := make([]error, 3)

for i, q := range parsed.ExpandedQueries[:3] {
    g.Go(func() error {
        r, err := searchSingle(gctx, q)
        if err != nil {
            errs[i] = err          // 개별 실패 기록 → 메트릭/로그에 반영
            return nil             // 의도적 nil: 형제 goroutine cancel 방지
        }                          // (errgroup은 non-nil 반환 시 ctx cancel)
        results[i] = r
        return nil
    })
}
_ = g.Wait()                       // nil만 반환되므로 err 사용 안 함

// 전체 실패 감지: 3개 모두 에러면 상위로 전파 (errors.Join는 Go 1.20+)
if errs[0] != nil && errs[1] != nil && errs[2] != nil {
    return nil, fmt.Errorf("all recall sub-queries failed: %w", errors.Join(errs...))
}
// dedup + merge (partial 허용)

구조적 주장: 순차 RTT 누적이 병렬로 압축될 수 있다. 실제 이득 크기는 Vault/enVector RTT 분포 · gRPC channel HOL blocking · HTTP/2 multiplex 효율에 의존하므로 벤치가 필요하다 (⑩.3). gRPC 채널은 goroutine-safe이므로 단일 채널로 병렬 호출 가능하다는 점은 확정된 사실.

핵심 구현 주의: errgroup은 non-nil 반환 시 gctx를 cancel해서 형제 goroutine을 조기 종료한다. 개별 실패를 "부분 결과로 계속"으로 처리하려면 반드시 return nil하고 에러를 외부 슬라이스에 기록해야 한다.

Go 1.25 대안 — sync.WaitGroup.Go: errgroup이 제공하는 "첫 에러 시 cancel" 세만틱이 필요 없고 순수한 fan-out만 원한다면 Go 1.25부터 stdlib sync.WaitGroup에 추가된 wg.Go(func())로 Add/Done 보일러플레이트 없이 작성 가능. 단, Recall은 errgroup의 ctx 연동이 필요해 그대로 두고, 더 단순한 사례(로그 라인 병렬 masking 등)에 적용.

7.3.2 임베딩 batch

Qwen3-0.6B ONNX는 batch dim 지원. expanded_queries[:3]을 단일 embed 호출로 묶으면 3회 inference → 1회로 축소된다. 절감 폭은 batch overhead · 모델 크기 · CPU 캐시 거동에 따라 달라지므로 벤치 필요 (⑩.3).

7.3.3 Cold start 특성

  • 데몬 수명 동안 모델 1회 로드 → 세션/요청별 cold start는 구조적으로 소멸 (정성적 주장)
  • 데몬 startup 자체는 모델 로드 시간에 좌우됨 — 실측 필요. 대응: /health → 503 {"status":"starting"}로 명시적 상태 표시
  • launchd/systemd로 OS 부팅 시 미리 기동하면 첫 요청 직전에는 준비 완료
  • 디스크 캐시 모델 mmap 사용 시 warm cache 이득 존재 여부는 벤치로 확인 (⑩.2)

7.3.4 Vault RTT 최소화 (post-MVP 가능성)

현재 recall은 Vault 왕복 2회 (DecryptScores + DecryptMetadata). 하나의 DecryptBundle RPC로 묶으면 왕복 1회로 줄일 수 있다. 절감 폭은 단일 Vault RTT 크기에 비례 — 벤치 기반으로 우선순위 판단. rune-admin proto 변경 필요 → 결정 #39로 post-MVP 추적.

7.4.1 Panic isolation + Go 1.22 routing

// Go 1.22+ stdlib ServeMux — method+path 라우팅 (gorilla/mux, chi 등 외부 router 불필요)
mux := http.NewServeMux()
mux.Handle("POST /capture",        recoverMW(captureH))
mux.Handle("POST /recall",         recoverMW(recallH))
mux.Handle("GET  /vault/status",   recoverMW(vaultStatusH))
mux.Handle("GET  /captures/{id}",  recoverMW(captureByIDH))  // path param
mux.Handle("POST /reload",         recoverMW(reloadH))

// 모든 HTTP handler 첫 줄 (middleware로 자동 적용)
func recoverMW(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        defer func() {
            if rec := recover(); rec != nil {
                slog.ErrorContext(r.Context(), "panic in handler",
                    "panic", rec, "stack", debug.Stack())
                writeError(w, 500, "INTERNAL_ERROR", "unexpected error")
            }
        }()
        next.ServeHTTP(w, r)
    })
}

한 handler 버그가 데몬 전체를 죽이지 않음. Go 1.22 stdlib mux는 method mismatch면 405, 미등록 path는 404를 자동 반환해 별도 router 라이브러리를 도입할 필요가 없다 — 외부 deps 하나 줄임. path param은 r.PathValue("id")로 접근.

7.4.5 Config reload 안전

  • fsnotify debounce 200ms (vim :w 등 에디터가 rename+write로 다중 이벤트 발생 — 100ms는 짧을 수 있음)
  • 파싱 실패 시 기존 config 유지 (fail-safe, 빈 config로 바뀌지 않도록)

  • SIGHUP 핸들러로도 reload 트리거 (launchd에서 보낼 때 유용)

1. No globals. 모든 것 주입

"Accept interfaces, return structs" (Rob Pike). Interface는 사용하는 쪽(internal/ports/)에서 정의, 구현은 internal/adapters/.

4. Thin transport layer

internal/transport/http/는 I/O만 — request 파싱, middleware, response 포맷. 비즈니스 로직 금지. 모두 internal/daemon/*/service.go로 위임.

5. Every port has a mock

각 ports.* interface는 같은 패키지에 Mock* 구조체를 제공. 테스트에서 한 줄로 주입. testify/mock 같은 framework는 도입 안 함 (stdlib testing만).

    Ready() bool
}

8.6a synctest 예시 — fsnotify debounce 200ms

// internal/daemon/config/reload_sync_test.go
//go:build go1.25   // synctest는 1.25부터 stable (1.24는 GOEXPERIMENT)

func TestDebounce(t *testing.T) {
    synctest.Test(t, func(t *testing.T) {
        loader := newLoader(200 * time.Millisecond)
        done := make(chan struct{})
        go func() {
            loader.Run()      // fsnotify 이벤트 처리 루프
            close(done)
        }()

        // 150ms 간격으로 3번 이벤트 → debounce로 1번만 reload 기대
        loader.events <- "write"
        synctest.Wait()       // 모든 goroutine이 blocked될 때까지 기다림
        time.Sleep(150 * time.Millisecond)
        loader.events <- "write"
        synctest.Wait()
        time.Sleep(150 * time.Millisecond)
        loader.events <- "write"

        // 300ms 대기 → debounce 발화
        time.Sleep(300 * time.Millisecond)
        synctest.Wait()
        require.Equal(t, 1, loader.reloadCount.Load())
    })
}

왜 중요한가: fsnotify/circuit breaker/retry는 "시간 의존성" 때문에 전통적으로 time.Sleep(500 * time.Millisecond)을 테스트에 박아야 했다 → flaky + 느림. synctest는 시계를 소스 레벨 수정 없이 가상화하고 "모든 goroutine이 blocked"되는 순간을 synctest.Wait()로 잡아낸다. 결과적으로 타이밍 테스트가 결정적이 되고 초당 수천 개 실행 가능.

8.7 Mock convention (handwritten, generator 없음)

전략: mock generator (gomock / mockery / moq)는 도입하지 않고 handwritten mock만 쓴다. 그러나 assertion library로 testify/{require,assert}는 사용 — 테스트 코드 ergonomics를 위한 실용적 타협.

모든 mock은 같은 패턴:

  • 구조체 이름: Mock<InterfaceName> (예: MockEmbedder)
  • 각 메서드마다 XxxFunc func(...) 필드 — 지정 시 그 동작, 없으면 default
  • 호출 카운트: XxxCallCount atomic.Int64 — spy 역할, race detector 대응
  • 기록 슬라이스 (필요 시): Entries []<type>

왜 generator 안 씀: (a) 코드 생성 diff로 리뷰 부담 추가, (b) stack trace가 generated code 경로로 가서 가독성 저하, (c) 100 LoC 미만이라 유지 부담 없음, (d) 팀원이 새 mock 만들 때 명확한 pattern 복제 가능.

왜 testify assertions 씀: require.NoError(t, err) (fatal)과 assert.Equal(t, want, got) (non-fatal)의 구분이 실질적 가치. 표현력 있는 실패 메시지 무료. stdlib `if err != nil { t.Fatalf(...) }`보다 의도 명확.

대안 (더 엄격한 선택): testify도 버리고 stdlib만 쓰려면 helper 함수 (mustNoErr(t, err), checkEqual(t, want, got))를 test/helper.go에 모아 사용. deps zero가 되지만 표현력은 다소 감소. 팀 선호에 따라 결정.

문제 상황

현재 Python rune은 sentence-transformers(PyTorch 기반)로 Qwen3-Embedding-0.6B를 실행. Go로 포팅하려면 동일 모델을 Go에서 돌릴 방법이 필요함. 그러나 Go 생태계에는 PyTorch 바인딩이 사실상 부재(gotch는 유지 중단 상태). 현실적 경로는 ONNX Runtime(언어 독립 실행 엔진) 또는 외부 delegate(Python subprocess / Ollama). 단일 바이너리·정적 배포 원칙을 지키려면 ONNX 쪽이 정답 후보.

자료 조사 / 사실 확인 (2026-04-17)
  • Qwen3-Embedding-0.6B ONNX export 존재 — onnx-community/Qwen3-Embedding-0.6B-ONNX (HF). 사용자 확인 완료
  • 기존 export의 범위 — base transformer만 포함. 입력이 input_ids, attention_mask, position_ids, past_key_values.{0..27}.{key,value}(KV cache 28 layer × 2)로 노출. 출력은 last_hidden_state. tokenize · pooling · normalization은 사용자 책임 (discussion #3의 Python 샘플 코드로 확인)
  • KV cache 입력이 노출된 이유 — HuggingFace transformers 기본 ONNX export가 use_cache=True(generation-ready)라 decoder-only 모델의 cache 인터페이스를 그대로 드러냄. embedding 용도에서는 dead weight
  • Python rune은 KV cache를 실질 사용 안 함 — mcp/adapter/embeddings.py:93-97의 SBERTSDKAdapter.get_embedding이 SentenceTransformer.encode() 단일 호출. 내부적으로 단일 forward pass만 수행
  • Pooling 전략 — Qwen3는 decoder-only(causal attention)라 encoder 모델의 mean pool 대신 last-token pool 사용. padding_side='left'가 전제
  • Go ONNX 바인딩: github.com/yalue/onnxruntime_go가 de facto. github.com/owulveryck/onnx-go는 pure Go지만 operator 지원 제한적 → 최신 모델 대부분 못 돌림
  • Go tokenizer: github.com/daulet/tokenizers가 HF Rust tokenizers를 CGO 래핑 → Python과 bit-identical 결과. 대안 github.com/sugarme/tokenizer는 pure Go 재구현이라 Qwen3 BPE 호환성 검증 필요
  • ONNX wrapper 재-export 가능 범위: PyTorch에서 torch.onnx.export에 nn.Module wrapper를 감싸면 position_ids 생성, use_cache=False(KV cache 입력 제거), last-token pool, L2 normalize를 그래프 안에 포함 가능. tokenizer만 구조적으로 불가능(문자열 파싱이라 ONNX 텐서 연산 아님)
검토한 선택지
  • (a) 기존 onnx-community export + Go 전부 구현: tokenize + KV cache 빈 텐서 초기화(28×2) + position_ids + last-token pool + L2 norm 모두 Go로 (~200-300 LoC). 부동소수점 순서 차이로 Python과 미세 결과 불일치 리스크
  • (b) PyTorch wrapper로 재-export + Go 최소 배선: pooling/norm/position_ids/no-KV-cache 를 그래프에 포함. Go 코드 ~50 LoC. Python(ONNX Runtime Python)과 Go가 같은 그래프를 호출하므로 결과 비트 일치
  • (c) Python subprocess delegate: 단일 바이너리 목표 포기. 의존성·배포 복잡도 증가
  • (d) Ollama HTTP delegate: 외부 서비스 의존. 오프라인 실패
최종 결정: (b) + 위의 1a-1g

왜 (b)를 선택했나:

  • Go 코드 대폭 감소 (~300줄 → ~50줄). last-token pool의 attention_mask 기반 인덱싱은 off-by-one/left-padding 처리 등 실수하기 쉬운 영역 — 이걸 PyTorch에서 처리하면 Python이 이미 검증한 로직을 그대로 사용
  • Python/Go 비트 일치: 같은 ONNX 그래프를 두 언어가 호출하므로 부동소수점 연산 순서까지 동일. 기존 Python으로 저장된 레코드 호환성 보장 (결정 #23과 연동)
  • KV cache 비대칭 해소: Python rune이 실질 사용하지 않는 KV cache를 Go만 빈 텐서로 억지 관리하는 건 비대칭. use_cache=False로 재-export하면 그래프에서 입력 자체가 사라져 대칭 회복
  • 유지 비용: scripts/export_qwen3_onnx.py 스크립트 한 번 작성·commit, Qwen 업스트림 업데이트 시에만 재실행. Go 코드는 모델 업데이트와 무관
  • (c)/(d) 비권장: 단일 바이너리 원칙 위배 + 유지 비용 증가. ONNX 실패 시나리오는 회피가 아닌 해결 대상

주의 사항 (Phase 0 실무):

  • wrapper export 시 use_cache=False 명시 + past_key_values 인자 제거
  • Qwen3-Embedding 모델 라이선스 확인 (Apache-2.0인 경우 자체 export 결과물 재배포 문제 없음)
  • ONNX Runtime 자체는 동적 라이브러리 (libonnxruntime.{so,dylib,dll}) → 배포 시 runed 바이너리 + 이 라이브러리 2-file bundle. 엄밀히 "단일 바이너리"는 아니지만 venv/pip/torch 대비 훨씬 단순

#4 ✅ 결정됨 Phase 0 암호화

AES-256-CTR 바이너리 포맷 — pyenvector와 bit-identical

최종 포맷 (Go에서 동일 재현):

  • 알고리즘: AES-256-CTR (pyenvector/utils/aes.py L52-58 실측)
  • 키: 32 바이트 (256비트) agent-specific DEK (Vault가 번들로 제공)
  • IV: 16 바이트 crypto/rand.Read로 **매 암호화마다 새로 생성**. counter 재사용 금지 (CTR 모드 IV 재사용은 동일 키로 암호화 시 keystream 노출)
  • Ciphertext 바이너리: IV(16B) || CT(=plaintext 길이). CTR은 stream cipher이므로 padding 불필요, 출력 길이 = plaintext 길이 + 16
  • Wire 인코딩: 위 바이너리 전체를 base64.StdEncoding으로 문자열화
  • Envelope JSON: {"a": "agent_xyz", "c": "<base64(IV||CT)>"} — 필드 순서 무관, JSON object. 결정 #46 A안 채택 시 "m" 필드(HMAC-SHA256) 추가
Go 구현 (참고): 
block, _ := aes.NewCipher(dek)          // 32B DEK
iv := make([]byte, 16)
_, _ = io.ReadFull(rand.Reader, iv)
stream := cipher.NewCTR(block, iv)
ct := make([]byte, len(plaintext))
stream.XORKeyStream(ct, plaintext)
combined := append(iv, ct...)
envelope := map[string]string{
    "a": agentID,
    "c": base64.StdEncoding.EncodeToString(combined),
}
상태: ✅ 바이너리 포맷 확정. Go crypto/cipher.NewCTR로 구현 가능. "IV random vs nonce" 미결은 해소됨 — random (crypto/rand)
📋 결정 로그 · 문제 상황 · 자료 조사 · 근거 (클릭 펼치기)
문제 상황

rune이 capture 시 metadata를 agent-specific DEK로 AES 암호화해서 envector에 저장. Vault의 DecryptMetadata RPC 또는 클라이언트(pyenvector/envector-go)가 같은 바이너리 포맷을 읽어야 함. Go로 포팅 시 pyenvector가 쓰는 것과 비트 단위로 같은 포맷을 내야 기존 저장 레코드가 호환됨 (결정 #23과 연결). 문서(pyenvector docstring)에 "AES-GCM"이라고 쓰여 있지만 실제 구현이 어느 모드인지 불명확 → 실측 필요.

자료 조사 / 사실 확인 (2026-04-17)
  • pyenvector 소스 직접 확인 (pyenvector/utils/aes.py L52-58):
    • cryptography 라이브러리 사용
    • Cipher(algorithms.AES(key), modes.CTR(iv))로 AES-256-CTR
    • IV 16 바이트 os.urandom(16)로 생성 → random
    • 결과: iv + ciphertext concat → base64.b64encode()
  • docstring의 "AES-GCM" 언급은 stale. git blame 상 오래된 주석이고 실 구현은 CTR. 초기 RFC 분석이 docstring에 의존해 "GCM"이라고 기술한 것이 혼란의 원인
  • CTR mode 특성:
    • stream cipher로 동작 → padding 불필요 (CBC/ECB와 다름)
    • 인증 없음 → 바이트 flip malleability 공격 가능 (결정 #46에서 HMAC 추가 검토)
    • 동일 키로 동일 IV 재사용 시 keystream 노출 → IV는 반드시 매회 random 또는 counter 순증
  • Envelope JSON 포맷: {"a": agent_id, "c": base64(IV||CT)}. 필드 2개뿐. 결정 #46 채택 시 "m" 추가 (HMAC-SHA256 앞 16B base64)
  • DEK 출처: Vault의 GetPublicKey 응답 번들에 agent-specific agent_dek(32B) 포함. runed가 메모리에 들고 있다가 encrypt/decrypt에 사용. 세부는 docs/runed/03-external-communication.md
  • Go stdlib 커버리지: crypto/aes.NewCipher + crypto/cipher.NewCTR + crypto/rand + encoding/base64로 전부 구현 가능. 외부 deps 0
검토한 선택지 (구현 관점)
  • (a) pyenvector 포맷 그대로 재현 (AES-256-CTR, IV random, no padding): 기존 레코드 비트 호환

  • (c) Fernet 등 high-level 래퍼 사용: Python과 포맷 불일치 → 호환 깨짐
최종 결정: (a) pyenvector CTR 포맷 그대로 재현

#46 블로킹 Phase 0→1 암호화

AES-256-CTR envelope에 MAC 추가 (malleability 차단)

현행 {"a":agent_id, "c":base64(IV||CT)}는 인증 없는 CTR이다. 공격자가 envelope의 IV 또는 CT 바이트를 flip하면 복호화 측이 이를 감지하지 못하고 임의 plaintext를 뱉을 수 있다. pyenvector와의 호환을 유지하면서 순차적으로 닫는 전략:

암호화/복호화 주체 명확화: AES-256-CTR envelope는 클라이언트 측(pyenvector / envector-go)에서 암·복호화된다. Vault는 FHE CipherBlock(DecryptScores/DecryptMetadata)만 담당하며 AES envelope 경로에는 관여하지 않는다. 따라서 업스트림 수정 범위는 pyenvector + envector-go 두 군데, Vault는 변경 없음.
옵션 A (추천, Phase 1 내): envelope 필드 "m" 추가 — HMAC-SHA256(dek, a || iv || ct)의 16B prefix. pyenvector·envector-go가 legacy(m 없음)와 신(m 있음) 두 모드 모두 decrypt 지원하도록 동시 패치. 기존 레코드는 m 필드 부재 시 verify skip (migration grace period 4주 후에는 경고 로그).
옵션 B (장기): crypto/cipher.NewGCM으로 마이그레이션. pyenvector 내부 포맷 자체 변경이라 legacy 레코드 재암호화 경로 필요 — 장기 로드맵.
옵션 C: 현상 유지 (비권장) — metadata malleability 취약 그대로 남음.
상태: 결정 필요. 영향 범위: pyenvector(Python) + envector-go(Go) — 둘 다 클라이언트 측. Vault와 FHE CipherBlock 경로는 무관. Phase 1 초반 A안 채택 → 두 SDK 동시 릴리스 필요.

⑩.1 Embedding 품질 & 런타임 벤치 (int8 채택 판단)

무엇을 확인: fp32와 int8 양자화 ONNX의 품질 차이 및 추론 속도.

절차:

  1. 같은 Qwen3-0.6B 모델의 fp32 / int8 ONNX 두 버전 준비
  2. 동일 코퍼스(실제 capture_log에서 샘플 N=최소 500개 선별, PII 마스킹 후)에서 각 벡터 생성
  3. Novelty 분류 일치율: 동일 (쿼리, 후보) 쌍에 대해 fp32·int8이 같은 novelty class (novel / related / near_duplicate)를 내는 비율
  4. Recall@K 순위 일치: 같은 쿼리로 top-5 recall 수행 후 두 버전의 Jaccard similarity 및 Kendall tau
  5. 추론 latency: 배치 크기 1·3·8로 각각 측정
  6. RSS: 모델만 로드한 최소 데몬 기준

판단: 결과 보고 팀이 "recall 품질 손실이 rune 용도에 허용 가능한가"를 합의 — 사전에 임계치를 commit하지 않는다. 결과를 보고 결정.

⑩.2 메모리 벤치 (steady-state / 멀티세션)

무엇을 확인: 단일 데몬 RSS가 어디에 얼마나 소비되는지, 세션 증가에 따른 선형성 여부.

시나리오:

  • Python baseline: 1 / 2 / 3 / 5 세션 동시 기동 → 각 PID의 RSS 합산. ps·pmap·/proc/pid/smaps로 모델/힙/키 분해
  • Go idle: runed 단독 기동 직후 RSS + 1시간 후 RSS (leak 감지)
  • Go under load: k6로 capture 1 req/s 1시간 지속 시 RSS 상한
  • Go 멀티세션: shim 1 / 2 / 3 / 5 spawn + runed 1개 → 총 RSS 변화. 세션 수와 무관한 상수 수렴 여부 검증
  • EvalKey 로드 방식 비교: ReadFile vs mmap에서 RSS 및 FHE 연산 latency 측정 → #37 판단 근거
  • goroutine leak: go.uber.org/goleak CI 게이트

보고 포맷: 시나리오별 RSS 그래프 + 세션 수 대비 RSS 기울기. Python 선형 vs Go 상수 수렴이 정량적으로 드러나야 "프로세스 모델 변경의 이득"을 commit 가능.

⑩.3 Latency 벤치 (capture / recall)

무엇을 확인: capture / recall의 p50·p95·p99 latency, 그리고 각 단계(embedding · Vault · enVector · rerank · serialize)가 전체에 얼마나 기여하는지.

시나리오:

  • Python baseline: 동일 워크로드 셋에 대해 capture/recall end-to-end p50/p95/p99 측정 (최소 iteration 100)
  • Go 단계별 분해: 각 phase를 slog trace로 기록해 Amdahl의 법칙 준수 여부 확인 (어느 단계가 병목인지)
  • Recall 병렬화 이득: expanded[:3]을 순차 처리하는 branch와 errgroup 병렬 branch를 A/B 비교. 이득이 기대보다 작으면 HOL blocking / HTTP/2 stream 한도 / Vault 서버 측 직렬화 원인 조사
  • Embedding batch: 단일 호출 ×3 vs batch-3 ×1 비교. ONNX Runtime Go의 batch dim 처리 효율 실측
  • Vault 단일 RTT: DecryptScores만 측정 → #39 DecryptBundle 우선순위 근거
  • Cold start: 데몬 fresh 기동 → 첫 capture까지 wall-clock

합격 기준: Python baseline 대비 회귀 없음이 최소 요건. 그 이상의 개선 폭은 벤치 결과를 보고 팀이 평가.

⑩.4 Shadow run / cutover gate (Scientist pattern)

무엇을 확인: Python(현행)과 Go(신규)가 동일 request에 대해 실제 프로덕션 트래픽에서도 일관된 결과를 내는지. 이게 통과해야 cutover 진행 (#48).

절차:

  1. 프로덕션 트래픽은 Python이 그대로 처리
  2. Go 데몬을 mirror로 붙여 동일 request 비동기 복제 — Go는 insert skip (이중 write 방지)
  3. 매 mirror마다 ~/.rune/logs/diff.jsonl에 기록:
    • capture: embedding vector cosine similarity · AES envelope byte-identical 여부 · novelty class 일치
    • recall: top-k id 집합 Jaccard · 순위 Kendall tau · score 차이 분포
  4. 기간 동안 Python-Go diff 분포를 histogram으로 수집

cutover 게이트: 수집된 diff 분포를 보고 팀이 허용 임계치를 결정 → 사전에 "0.1% 미만" 같은 숫자 commit하지 않는다. 실측 분포가 bimodal이면 (대부분 0에 가깝고 소수가 큰 차이) 그 outlier의 원인을 먼저 진단해야 함. Go 쪽 panic/OOM이 Python으로 번지지 않는지 shadow run 중 지속 확인.

⑩.5 부하 / 장애 시나리오 테스트

무엇을 확인: 정상 상태 성능만큼이나 장애 상태에서의 동작이 예측 가능한지 (#40·#51 연관).

  • 부하 테스트: k6로 1/10/50 req/s 유지 시 p99 및 RSS 추이 (멀티세션 폭주 backpressure)
  • Vault 장애 시나리오: Vault를 일시 중단 → 2-retry 후 retryable error 반환 검증 (#40). breaker 개방 → fast-fail 전환 검증 (#51)
  • Vault 플랩: 1초 up/down 반복 → breaker half-open probe 정상 동작 검증
  • Sleep/wake: 실제 macOS caffeinate/pmset로 sleep 유도 → wake 후 다음 capture 성공까지 시간
  • 데몬 크래시: kill -9 → launchd/systemd 자동 재시작 + 첫 요청까지 시간
  • Config 손상: config.json에 malformed JSON 주입 → 기존 cfg 유지 + 로그 경고 확인
  • OOM 근접: GOMEMLIMIT 아래에 큰 부하 → GC 공격적 진입 확인, OOM kill 발생 여부

⑩.6 정확성 회귀 (golden fixture)
